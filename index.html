DISKO 1 — Disk Image Flag Recovery
picoCTF · Disk Forensics · Linux CLI
A raw disk image stores all data sequentially — including deleted files and plaintext strings — without needing the filesystem to be mounted. By extracting all printable ASCII sequences and filtering for the flag pattern, the flag can be recovered directly from binary data without a full forensic mount.
disk forensics strings grep linux cli
  • Why decompress first? gunzip restores the raw .dd image — strings cannot parse compressed binary correctly, so decompression is required before any analysis.
  • Why strings? It scans byte-by-byte for sequences of printable characters (≥4 chars by default), surfacing human-readable data embedded anywhere in the binary — regardless of filesystem structure.
  • Why pipe to grep? The output of strings can be thousands of lines. Piping to grep picoCTF filters only lines matching the known flag format, reducing noise instantly.
# Decompress the disk image gunzip disko-1.dd.gz # Extract all printable strings, filter for flag pattern strings disko-1.dd | grep picoCTF

WEB EXPLOIT
SSTI — Server-Side Template Injection via Jinja2
picoCTF · Web Exploitation · Remote Code Execution
When user input is embedded directly into a template engine without sanitization, the server evaluates it as code rather than data. This breaks the boundary between input and execution — a fundamental security failure that escalates from data reflection to full Remote Code Execution on the host.
picoCTF{s4rv3r_s1d3_t3mp14t3_1nj3ct10n5_4r3_c001_3066c7bd}
SSTI Jinja2 RCE web exploitation
  • Why inject {{7*7}}? Template engines evaluate expressions inside {{}}. If the server returns 49 instead of the literal string {{7*7}}, it confirms the input is being executed server-side — proof of SSTI.
  • Why use the Jinja2 context chain? Jinja2 sandboxes direct access to os, but the template context object retains references to Python built-ins. Traversing cycler.__init__.globals reaches the os module without triggering sandbox restrictions.
  • Why popen().read()? os.popen() opens a shell subprocess and returns its output as a file object — .read() captures stdout back into the template, making the result visible in the HTTP response.
# Confirm SSTI — server should return 49, not the literal string {{7*7}} # RCE via Jinja2 context chain to bypass sandbox {{ self._TemplateReference__context.cycler.__init__.globals.os.popen('cat flag').read() }}
✓ Solved
WEB EXPLOIT
Cookie Monster — Sensitive Data in Browser Cookie
picoCTF · Web Exploitation · Cookie Analysis
Cookies are client-side storage — the browser holds them and sends them with every request. Storing secrets in cookies, even Base64-encoded, is not encryption: Base64 is an encoding scheme with no key, trivially reversible by anyone who can read the cookie value via DevTools or HTTP interception.
picoCTF{c00k1e_m0nster_l0ves_c00kies_c430AE20}
cookie analysis base64 CyberChef devtools
  • Why DevTools → Application → Cookies? Browsers expose all cookie key-value pairs in plaintext via the Application tab. No proxy or interception needed — if a cookie is set, it's readable here.
  • Why Base64, not encryption? Base64 encodes binary data as ASCII for safe transport — it has no secret key and provides zero confidentiality. Anyone with the encoded string can decode it instantly. Developers sometimes mistake it for obfuscation.
  • Why remove non-alphabet chars first? URL-encoded cookies may contain %3D (padding =) or spaces that break standard Base64 decoders. Stripping non-alphabet characters normalizes the string before decoding.
✓ Solved
BINARY EXPLOIT
Pie Time — Bypassing PIE via Leaked Base Address
picoCTF · Binary Exploitation · GDB · PIE Bypass
PIE (Position Independent Executable) randomizes a binary's load address at runtime, making hardcoded jump targets useless. However, if any runtime address leaks — even just main() — the entire binary's layout is recoverable, since relative offsets between functions remain fixed regardless of where the binary is loaded.
picoCTF{b4s1c_p051t10n_1nd3p3nd3nc3_f8845f06}
PIE bypass GDB binary exploitation offset calculation
  • Why disassemble locally with GDB? The binary's static layout — addresses before ASLR is applied — is fixed on disk. GDB reveals the compile-time addresses of main and win, so their offset can be calculated without running the binary on the target server.
  • Why does one leaked address break PIE? PIE shifts the entire binary by a single random base offset. All functions shift by the same amount. So: win_runtime = main_runtime − (main_static − win_static). One leak defeats the entire randomization.
  • Why subtract, not add? main sits at a higher address than win in this binary. The offset is 0x60 — subtracting it from the leaked main address gives the exact runtime location of win().
# Get static offsets from local binary (gdb) disassemble main # note first instruction address (gdb) disassemble win # note first instruction address # offset = main_static - win_static = 0x60 # win_runtime = leaked_main_runtime - 0x60
✓ Solved
